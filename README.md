# Fable Poker — GTO ソルバー

ブラウザで動くヘッズアップ・ポストフロップ GTO ソルバー(GTO Wizard 風)。
Rust 製 CFR+ エンジンを WebAssembly にコンパイルし、Web Worker 上で実行します。

## 構成

```
solver/   Rust ソルバー (依存クレートゼロ / wasm-bindgen 不使用の素の C ABI)
web/      React + Vite + TypeScript フロントエンド
```

### ソルバー (solver/)

- **アルゴリズム**: CFR+ (regret-matching+、交互更新、線形平均)
- **16bit量子化ストレージ**: regret/累積戦略をノード毎スケール付き u16 で保持し
  メモリを半減 (戦略はハンド毎の正規化でスケールが消えるため生の u16 のまま計算)
- **バッファプール**: 探索中のヒープ割り当てゼロ
- **マルチスレッド (分割ソルブ)**: 最初のチャンスノード(フロップ解析ならターン
  カード)以下のサブツリーが互いに独立なことを利用し、サブツリーを K 個の Web
  Worker に分配。各反復は「メイン: 開始ストリートのリーチ計算 → ワーカー並列:
  サブツリー CFR → メイン: 値を回収して更新」のフォークジョイン。
  SharedArrayBuffer 不要 (COOP/COEP ヘッダなしで動作)、安定版 Rust のみ。
  クラスタ=単一ソルバーの一致はネイティブテストで検証
- **WASM SIMD** (+simd128) 有効
- **ソルブ結果のキャッシュ**: 解いたスポット (ボード×レンジ×ツリー設定が同一)
  は IndexedDB に保存され、次回は即座に復元。約 2GB を超えると古いものから削除
- **PWA / オフライン対応**: Service Worker が初回訪問時に全アセット (wasm 含む)
  をプリキャッシュ。2回目以降は完全オフラインで起動・ソルブ可能。スマホでは
  「ホーム画面に追加」でアプリとして使える (要 HTTPS または localhost)
- **モバイル対応**: レスポンシブレイアウト、タッチでのレンジペイント。
  フロップ全体の解析はメモリの小さい端末では失敗する場合あり (ターン/リバー推奨)
- **ショーダウン評価**: リバーごとにハンドランクを事前ソートし、カードリムーバル込みの O(n) スイープで全コンボの勝敗を一括計算
- **スート同型性**: ターン/リバーのチャンスカードを盤面を固定するスート置換で同値類にまとめ、代表カードのみ解く(モノトーン/ツートーン盤で計算量を大幅削減)。非正準カードの照会は置換合成でマッピング
- **搾取可能度**: ベストレスポンス計算で収束を実測 (% of pot)
- **EV / エクイティ**: ノード単位でオンデマンド計算

### 機能

- プリフロップラインビルダー: オープン位置 (EP/HJ/CO/BTN/SB) × 相手位置 ×
  ポットタイプ (SRP / 3BET / 4BET) を自由に選択。ポット・スタック・両者の
  近似 GTO レンジを自動生成 (すべて編集可)
- **EV 表示は GTO Wizard と同じ規約**: その決定時点から先の純増減 (bb)。
  それまでの自分の投資はサンクコストとして扱い、フォールド = 0 EV。
  ポラライズドリバーの解析解との一致をテストで担保
- 13x13 レンジエディタ (ドラッグペイント、ウェイト、レンジ文字列インポート)
- フロップ/ターン/リバー任意の開始ストリート
- 戦略マトリクス (アクション頻度の色分け、リーチ加重)
- コンボ別詳細 (頻度 / EV / エクイティ)
- ゲームツリーのナビゲーション (アクション・カード選択、パンくず)
- 解析中の途中経過閲覧 (反復を重ねるほど精度向上)

## ビルド & 実行

前提: Rust (wasm32 ターゲット), Node.js

```sh
# 1. wasm ターゲット (初回のみ)
rustup target add wasm32-unknown-unknown

# 2. ソルバーを wasm にビルドして web/public に配置
cd solver
cargo build --release --target wasm32-unknown-unknown
cp target/wasm32-unknown-unknown/release/solver.wasm ../web/public/solver.wasm

# 3. フロントエンド
cd ../web
npm install
npm run dev
```

## スマホで使う / デプロイ

ビルド成果物 (`web/dist/`) は完全に静的なファイル群で、サーバーサイド処理も
特殊ヘッダ (COOP/COEP) も不要です。任意の静的ホスティングにそのまま置けます:

```sh
cd web && npm run build   # dist/ が完成品 (Service Worker のプリキャッシュ込み)
```

- **GitHub Pages / Cloudflare Pages / Netlify など**: dist/ をアップロードするだけ。
  HTTPS になるので PWA (ホーム画面に追加・オフライン動作) がフルに機能します
- **同一Wi-Fiで手軽に試す**: `npm run dev -- --host` で Mac の IP に
  スマホからアクセス (この場合 http なのでオフライン機能だけ無効)

初回アクセス時に全アセット (約 350KB) がプリキャッシュされるため、以降は
機内モードでも起動・ソルブ・キャッシュ済みスポットの閲覧がすべて動きます。

```sh
cd web && npm run test:mobile   # モバイル+オフラインの E2E
```

## テスト

```sh
cd solver && cargo test          # ソルバー単体テスト (クラスタ=単一の一致検証含む)
cd web && npm run test:smoke     # wasm C ABI スモークテスト (Node)
cd web && npm run test:cluster   # 分散プロトコルのスモークテスト (Node)
cd web && npm run test:ui        # Playwright E2E (要: npx playwright install chromium)
cd web && npm run test:cache     # キャッシュ復元の E2E
```

検証内容: 7カード役評価の境界ケース、同型類のカバレッジ、搾取可能度の単調減少、
EV のゼロサム性、ポラライズドリバーのブラフ/コール頻度・EV の解析解一致など。

`npm run compare:gtow` は実スポット (SB vs EP 3BP, Qd8s6c) を GTO Wizard の
公表値と突き合わせる比較スクリプト。EV は ±0.2bb、レンジ集計頻度は数%以内で
一致する (混合戦略の無差別ゾーンでは頻度の split 自体は一意でないため、
EV の一致が本質的な検証になる)。

ベンチマーク: `cd solver && cargo run --release --example bench [ボード]`

## 性能の目安 (M系Mac、ブラウザ内 6 ワーカー実測)

| スポット | メモリ | 1反復 (BR込み) | 搾取可能度 < 3% pot まで |
|---|---|---|---|
| ターン開始 (フルレンジ) | ~15 MB | ~0.01s | 1 秒未満 |
| フロップ全体・超軽量ツリー (SRP・レインボー) | ~560 MB | ~0.7s | ~15 秒 |

一度解いたスポットは IndexedDB キャッシュから即座に復元されます。
ツリー設定はフロップ解析なら「超軽量」(ターン/リバーのレイズなし) 推奨。
搾取可能度を見ながら途中で一時停止して閲覧できます。

## 制限事項

- ポストフロップはヘッズアップのみ (6-max はプリフロップレンジとして反映)
- レンジは 169 ハンドクラス粒度 (コンボ単位のウェイトは未対応)
- ICM / レーキ / マルチウェイ未対応
- プリセットレンジは近似値 (編集可能)
